Introduction

The product consists of the client and server side of a ride sharing service. There are two types of users, vehicles and passengers, that are classified by their username. Each type uses the service to post their location, and get the locations of all users of the other type. The purpose of this investigation was to determine what some of the significant security vulnerabilities that existed in the product.

Methodology

I focused on providing the server with problematic data, through curl on the command line, and observing whether the resulting effect could have been problematic. Some issues, which could permit attackers with malicious intent to utilize the product in a way that it was not intended, have been described below.

Issues found

  • Issue: Broken Authentication/ Sensitive Data Exposure
  • Location: POST /rides API
  • Severity: High
  • Description: Despite not being a driver, I am able to act as one by setting the username that I send to the server as a driver's (JANET in this case). This gives me access to the position data of all passengers, even though I am not actually a driver. If an app like Uber used this implementation, all an attacker has to do is correctly guess a driver's username, to gain access to sensitive data from recent passengers.
  • Proof: Screenshot shows data for numerous passengers being sent to server, followed by data for JANET
  • Resolution: Add user authentication to the site. Would be best to use existing framework, such as Passport.js.
  • Issue: Cross Site Scripting
  • Location: POST /rides API and GET / (home)
  • Severity: High. XSS vulnerabilities are often preyed upon by attackers. They allow often attackers manipulate client side Javascript, by submitting strings of code within POST requests.
  • Description: Javascript commands (sent to rides API as data) are accepted by the server and inserted into database in the exact format and syntax that they were received in
  • Proof:

    The first screenshot shows Javascript being sent to server, as "username," using curl in the Terminal. Server accepts string and inserts into database. The second screenshot, of mLab window, shows document containing Javascript in database collection. The final screenshot shows the alert window, which was encoded in the XSS request, opened as the home page tries to display passenger list.

  • Resolution: Insert following line of code between lines 57 and 58 of index.js to sanitize input and remove special characters from username sent by client
    username = username.replace(/[^\w\s]/gi, '');
  • Issue: MongoDB Injection
  • Location GET /vehicles.json API
  • Severity: High
  • Description: This API is supposed to return data on the specific username given, if it exists. If $gt is added to the username requested (see screenshots for syntax), all entries in the database are greater than the query string username are returned. Thus, an attacker can access all documents in the vehicle collection, by querying with an empty username
  • Proof: Screenshot shows curl query that returns all driver's data
  • Resolution: Implement the same line of code for input sanitization, that is shown for the XSS problem above.
    username = username.replace(/[^\w\s]/gi, '');

Conclusion

All web apps, even those that sucessfully fulfill their purpose, are vulnerable to security threats. Developers should always integrate methods to alleviate the effect of potential attacks by users with malicious intent. Key inclusions, such as input sanitization, can help avoid a number of types of attacks, and improve the general security of the app.
For this app, I would recommend further investigation into improving security of the database and also into how client side processes can be used to make attacks more difficult. With that being said, I would recommend that all methods that protect against attacks be implemented on the server, so that malicious clients may not have access to, understand, or change code.
